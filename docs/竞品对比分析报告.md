## 一、项目概况

### 技术栈

| 层级 | 技术选型 | 说明 |
|------|----------|------|
| 桌面容器 | PyWebView | 轻量级原生窗口封装，启动快、资源占用低 |
| 前端框架 | Vue 3 + TypeScript + Naive UI + Vite | 现代化响应式界面，组件化开发 |
| 后端 API | FastAPI (异步) + WebSocket + SQLite (aiosqlite) | 高性能异步接口 + 双向实时通信 |
| 模型调用 | openai 库 | 统一兼容 OpenAI / Ollama / LM Studio 等主流协议 |
| 工具扩展 | MCP SDK (fastmcp) | 支持 stdio / SSE / streamable-http 三种传输方式 |
| 技能系统 | 受限 exec 沙箱 + PyYAML + zipfile | prompt / code 双类技能，SKILL.md 包导入导出 |
| 知识库检索 | sqlite-vec + FTS5 + numpy | 向量语义检索 + 全文关键词搜索，全部本地 SQLite |
| 文档解析 | markitdown + python-docx + PyPDF2 + pandas | 多格式内容提取 |
| 渲染增强 | marked + highlight.js + mermaid + KaTeX | 完整 Markdown 生态 |
| 语音交互 | Web Speech API | 浏览器原生语音输入/朗读，零 API 依赖 |

### 核心功能清单

- **多角色/Profile 系统**：定义人格、Prompt、工具白名单、Skill 绑定，一键切换
- **MCP 工具生态**：stdio / SSE / streamable-http 三种传输，按角色隔离工具
- **Skills 技能系统**：prompt 型（指令+工具白名单）+ code 型（受限沙箱 Python），热重载，.zip 导入导出
- **知识库 RAG**：Obsidian 兼容 `[[双链]]` + 知识图谱 + FTS5 全文 + sqlite-vec 语义混合检索 + 标签系统
- **Human-in-the-Loop**：`system_ask_user` 提问 + 工具审批流 + WebSocket 实时回传
- **Agent 自主规划**：`system_todo` 任务规划 + `system_delegate_task` 子智能体委派
- **流式工具执行**：边生成 tool_call 边执行，不等全部生成完
- **并行工具调用**：`asyncio.gather()` 并行执行独立工具
- **模型容错降级**：指数退避重试 + fallback 备用模型自动切换
- **Prompt Caching**：系统提示词+工具定义按序排列，最大化前缀缓存命中
- **上下文自动压缩**：长对话超阈值自动生成摘要
- **WebSocket 双向通信**：逐 token 推送 + 即时取消 + 审批实时回传
- **语音交互**：浏览器原生 SpeechRecognition + speechSynthesis
- **跨会话记忆**：AI 回复自动向量化，新对话语义召回
- **对话分叉**：从任意历史消息处分支新对话
- **会话导出/导入**：Markdown / JSON / ZIP 三种格式
- **主题自定义**：取色器 + 圆角半径 + 字号，实时预览
- **统计仪表板**：Token 消耗趋势图、对话统计
- **网页抓取→知识库**：`system_web_fetch` 抓取网页保存为笔记
- **自动增量索引**：文件监听 + 30s 内自动索引 + 5s 去抖合并

## 三、LumNeo 的六个独特护城河 🏆

### 1. 桌面原生 + 纯本地数据

所有数据在本地 SQLite（sqlite-vec + FTS5），零网络依赖。PyWebView 封装为桌面应用。

- **对比**：Dify/Coze/n8n 需要 Docker 服务器部署；AnythingLLM 虽有桌面版但基于 Electron（更重）；Open WebUI 纯 Web。
- **优势**：隐私最高等级，离线可用，启动快，资源占用低。

### 2. Skills 技能系统（对齐 Claude Agent Skills 协议）

```
prompt 型 → 指令注入 + 工具白名单展开
code  型 → 受限沙箱 Python，可被 LLM 作为 function 调用
热重载   → 编辑即时生效，无需重启
.zip 包  → SKILL.md + skill.py 导入导出
身份分级 → admin 可建 code 型，user 仅 prompt 型
```

- **对比**：OpenClaw 有 Skills 系统（2000+ 社区技能 + eBPF 安全），但非 Claude Skills 协议对齐；Dify 有"工具"但无可编程技能包；Coze 有插件市场但不可本地自定义执行。
- **唯一将 Claude Agent Skills 协议落地为桌面应用的项目。**

### 3. Obsidian 兼容知识库 + AI RAG 融合

```
[[双链]] 补全跳转 + 反向链接 → Obsidian 兼容
力导向知识图谱可视化
FTS5 全文搜索 + sqlite-vec 语义向量 = 混合检索
标签系统多维分类
网页抓取 → 自动入库
```

- **对比**：RAGFlow 文档解析最强但非笔记工具；MaxKB 知识库最结构化但无双链；FastGPT 有 RAG 但图谱弱；Obsidian 有双链但无 AI RAG。
- **唯一同时具备 Obsidian 笔记体验 + AI 语义检索 + 知识图谱的产品。**

### 4. Human-in-the-Loop 双层机制（聊天内原生嵌入）

```
system_ask_user    → AI 主动暂停向用户提问
工具审批流         → 敏感工具调用前拦截 → WebSocket 实时回传
```

- **对比**：AutoGen 的 HITL 需编写代码；Dify 的审批在工作流级别而非聊天内；Vercel AI SDK 7 的 `toolApproval` 是前端 SDK 级别。
- **在聊天上下文内自然嵌入审批，体验最流畅。**

### 5. 流式工具执行 + 对话分叉

- **流式工具执行**：模型边生成 tool_call 边执行，不等全部生成。主流平台多为"全部生成→批量执行"。
- **对话分叉**：从任意历史消息处分支新对话。Dify/Coze/Open WebUI 均无此功能。

### 6. MCP 协议全栈支持

stdio / SSE / streamable-http 三种传输方式全覆盖，且按角色隔离工具白名单。

- **对比**：AnythingLLM 最近支持 MCP；Open WebUI 通过 Pipelines；但都没有按角色粒度隔离。

## 四、与各竞品的详细对比

### 4.1 vs Dify (62k ⭐)

**Dify 定位**：国内最活跃的开源 AI 应用平台，LLMOps 一体化，拖拽构建 + 开源私有化。

| Dify 优势 | LumNeo 现状 | 差距程度 |
|-----------|------------|---------|
| 可视化拖拽工作流编辑器 | 聊天式 Agent 自主规划 | 🔴 大 |
| LLMOps（日志/追踪/标注/成本分析） | ✅ Span 追踪 + Trace Panel 瀑布图 + 统计仪表板 | 🟡 中 |
| 工作流发布为 API | 无 | 🔴 大 |
| 多租户 + RBAC | 单用户桌面 | 🟡 中（定位差异） |
| 批量测试/评估框架 | 无 | 🟡 中 |
| 50+ 内置工具 + OpenAI Plugin 兼容 | MCP 生态（标准协议） | 🟢 持平 |
| 数据集标注与评估 | 无 | 🟡 中 |
| 模板/应用市场 | 无 | 🟡 中 |

**可借鉴点**：
1. LLMOps 的日志追踪和标注反馈体系
2. 工作流节点的可视化状态展示（pending/running/done/error）
3. 数据集标注 + 批量评估框架

### 4.2 vs AnythingLLM

**AnythingLLM 定位**：全能 AI Agent 盒子，桌面 + Docker 双形态，RAG + Agent + MCP。

| AnythingLLM 优势 | LumNeo 现状 | 差距程度 |
|-----------------|------------|---------|
| No-code Agent Builder | Skills 系统（需要懂 Markdown/Python） | 🟡 中 |
| 智能 Skill 选择（节省 80% token） | Skills 全量注入系统提示词 | 🟡 中 |
| 移动端 App (Android) | 无 | 🟡 中 |
| 定时 Cron 任务 | 无 | 🟡 中 |
| 文档生成 Agent（PDF/Excel/PPTX） | system_tools 有基础文件操作 | 🟢 持平 |
| 30+ LLM 提供商 | OpenAI 兼容协议（也广泛） | 🟢 持平 |
| 9+ 向量数据库支持 | 仅 sqlite-vec | 🟢 持平（本地优先） |

**可借鉴点**：
1. **智能 Skill 选择**：按语义匹配只注入相关 Skill，大幅减少系统提示词 token 消耗
2. **Cron 定时任务**：让 Agent 在指定时间自动执行任务

### 4.3 vs Open WebUI

**Open WebUI 定位**：最强自托管 ChatGPT 替代品，统一前端接入多 LLM 后端。

| Open WebUI 优势 | LumNeo 现状 | 差距程度 |
|----------------|------------|---------|
| 团队协作（Channels/RBAC/SSO/SCIM） | 单用户 | 🟡 中（定位差异） |
| 9 种向量数据库支持 | 仅 sqlite-vec | 🟢 持平（本地优先） |
| 插件系统（Pipelines + 沙箱子解释器） | Skills 系统 | 🟢 持平（各有千秋） |
| 多模态输入（图片/音频） | 拖拽图片解析 + 语音 | 🟢 持平 |
| 多用户管理 + SSO | 单用户 | 🟡 中（定位差异） |

**可借鉴点**：
1. Pipelines 插件系统的热插拔机制
2. Channels 团队协作模式（如果未来扩展）

### 4.4 vs RAGFlow (18k ⭐)

**RAGFlow 定位**：最强 RAG 引擎，深度文档理解专家。

| RAGFlow 优势 | LumNeo 现状 | 差距程度 |
|-------------|------------|---------|
| OCR + 复杂表格提取 + 文档布局分析 | ✅ kb_parser.py：Tesseract OCR + pdfplumber 表格 + 布局感知分块 | 🟢 基本持平 |
| 多种分块策略（按标题/表格/语义） | ✅ 保留表格/代码块完整性 + 标题层级分块 | 🟢 基本持平 |
| 深度引用溯源（回答→源文档位置） | ✅ citation_id + 点击跳转源文件 | 🟢 基本持平 |
| 重排序 (Rerank) | 有基础支持 | 🟡 中 |
| 关键词+向量+重排序三重检索 | 语义+关键词双检索 | 🟡 中 |

**可借鉴点**：
1. **OCR 引擎**：集成 Tesseract/PaddleOCR 处理扫描件
2. **表格结构化提取**：参考 RAGFlow 的 deepdoc 解析器
3. **引用溯源高亮**：检索结果标注源文档位置，回答中可点击跳转

### 4.5 vs CrewAI (47k ⭐) + LangGraph (131k ⭐)

**CrewAI 定位**：最直觉的角色扮演多 Agent 框架。
**LangGraph 定位**：状态图编排引擎，精确控制 Agent 流程。

| CrewAI/LangGraph 优势 | LumNeo 现状 | 差距程度 |
|----------------------|------------|---------|
| 角色扮演多 Agent（role + goal + backstory） | ✅ crew.py：single/sequential/parallel 模式 + 角色定义支持 | 🟢 基本持平 |
| Agent 间直接对话/争论 | 子 Agent 独立上下文，通过链式模式传递结果 | 🟡 中 |
| 状态图编排（条件分支/并行/循环/中断恢复） | 线性 Agent 循环 + 并行扇出 | 🟡 中 |
| LangGraph checkpoint 持久化 | SQLite 持久化消息 + Span 追踪 | 🟢 持平 |
| LangGraph 600+ 集成 | MCP 生态（开放标准） | 🟢 持平 |

**可借鉴点**：
1. **角色化子 Agent**：role + goal + backstory 定义，让委派的子 Agent 有明确人格
2. **协作流程模板**：预置 research→write→review 等经典链式流程
3. **Agent 间消息传递**：子 Agent 之间可以对话和争论

### 4.6 vs MaxKB

**MaxKB 定位**：最成熟的企业知识库平台，Python/Django + Vue.js + PostgreSQL/pgvector。

| MaxKB 优势 | LumNeo 现状 | 差距程度 |
|-----------|------------|---------|
| DAG 工作流引擎（50+ 节点） | 无可视化工作流 | 🔴 大 |
| BERT 意图分类器（92% 准确率） | 无意图识别路由 | 🟡 中 |
| 知识库层级模型（知识→文档→段落→问题） | 文件级知识库 | 🟡 中 |
| P99 < 500ms 检索延迟 | 本地 SQLite 检索（也很快） | 🟢 持平 |
| 自动标签（TF-IDF + TextRank） | 手动标签 | 🟡 中 |
| 模板市场 | 无 | 🟡 中 |

**可借鉴点**：
1. **意图识别路由**：根据问题类型自动选择检索策略
2. **自动标签**：TF-IDF + TextRank 自动提取关键词为标签
3. **知识库层级模型**：知识→文档→段落→问题的结构化组织

### 4.7 vs OpenClaw (347k ⭐)

**OpenClaw 定位**：最雄心勃勃的开源 AI Agent 中枢，连接 13+ 即时通讯渠道。

| OpenClaw 优势 | LumNeo 现状 | 差距程度 |
|--------------|------------|---------|
| 13+ 即时通讯渠道 | 仅桌面应用 | 🟡 中（定位差异） |
| Memory Palace 3D 空间记忆 | 跨会话向量记忆 | 🟡 中 |
| eBPF 内核级 Skill 安全 | 受限 exec 沙箱 | 🟡 中 |
| ClawHub 2000+ 社区技能 | Skills 系统（自建） | 🟡 中 |
| People Wiki 关系图谱 | 无 | 🟡 中 |
| 移动端 App (iOS + Android) | 无 | 🟡 中 |

**可借鉴点**：
1. **社区技能市场**：长期可考虑 Skills 分享平台
2. **People Wiki**：为知识库增加"人物"维度的关系管理

### 4.8 vs LangSmith（互补品）

LangSmith 是 LangChain 的商业可观测性平台，**不是竞品而是互补品**。

| LangSmith 优势 | 可借鉴方向 |
|---------------|-----------|
| Span 级追踪（每步 tool call 输入/输出/耗时） | Agent 执行追踪 |
| 瀑布图可视化 Agent 执行流程 | 工具调用瀑布图 |
| SmithDB（Rust, P50 92ms 加载） | 高性能追踪存储 |
| LangSmith Engine 自主诊断 + 自动 PR | 智能错误分析 |
| 在线+离线评估器 | Agent 质量评估 |
| 标注队列 + 人类反馈 | 反馈收集系统 |
| PII/密钥检测和脱敏 | 安全增强 |

### 4.9 vs Vercel AI SDK（互补品）

Vercel AI SDK 是前端库（React/Vue/Angular），**不是竞品而是可参考的设计模式**。

| Vercel AI SDK 优势 | 可借鉴方向 |
|-------------------|-----------|
| useChat 状态机（submitted→streaming→ready→error） | 前端聊天状态管理 |
| Message Parts 架构（text/tool-call/tool-result/reasoning/file/source） | 消息数据结构 |
| Generative UI（`<JSXPreview />`） | 动态组件渲染 |
| MCP Apps（MCP 工具返回交互式 UI） | MCP 工具的前端渲染 |

## 五、Agent 框架横向对比

| 维度 | **LumNeo** | **CrewAI** | **LangGraph** | **AutoGen** | **MetaGPT** |
|------|-----------|-----------|--------------|-------------|-------------|
| **Stars** | — | 47k+ | 131k+ | 56k+ | 66k+ |
| **架构模式** | 聊天式 Agent Loop | 角色扮演 Crew | 状态图 (DAG) | 对话驱动 | SOP 驱动 |

| **状态管理** | SQLite 消息 | 内存 | ⭐⭐⭐⭐⭐ checkpoint | 内存 | 环境消息 |
| **Human-in-the-Loop** | ⭐⭐⭐⭐⭐ 原生双重 | ⭐⭐⭐ | ⭐⭐⭐⭐ 中断恢复 | ⭐⭐⭐⭐ | ⭐⭐ |
| **工具生态** | MCP 标准 | LangChain 兼容 | 600+ 集成 | MCP 原生 | 有限 |
| **生产就绪** | ✅ 桌面应用 | ✅ | ✅ + Temporal | ⚠️ 维护模式 | 🟡 |
| **最佳场景** | 个人桌面助手 | 内容/研究管道 | 复杂状态工作流 | 多 Agent 讨论 | 软件项目生成 |

### 架构模式对比

| 模式 | 代表框架 | 核心思想 | 优点 | 缺点 |
|------|---------|---------|------|------|
| **聊天式 Agent Loop** | LumNeo | LLM 自主决策→工具调用→结果反馈→继续 | 自然直觉，用户体验好 | 复杂流程控制弱 |
| **状态图** | LangGraph | 有向图 + 条件分支 + 状态持久化 | 精确控制，可观测 | 学习曲线陡 |
| **角色扮演** | CrewAI | 定义角色 + 任务 + Crew | 直觉，易上手 | 深度定制受限 |
| **对话驱动** | AutoGen | Agent 间消息传递 | 灵活，自然 | 输出不可预测 |
| **SOP 驱动** | MetaGPT | 模拟团队 SOP | 结构化输出 | 非软件场景不适用 |

## 七、优先级排序：最值得做的 5 件事

### P0 — 核心竞争力突破

#### 1. RAG 文档解析增强（对标 RAGFlow）

| 现状 | 目标 |
|------|------|
| markitdown + PyPDF2 基础解析 | OCR 扫描件识别 + 复杂表格提取 + 文档布局分析 |
| 无引用溯源 | 检索结果标注源文档位置，回答中可点击跳转原文 |
| 基础分块（按标题+窗口） | 多种分块策略（按表格/按语义/按文档结构） |

**技术路径**：
- 集成 Tesseract OCR / PaddleOCR 处理扫描件和图片
- 参考 RAGFlow 的 deepdoc 解析器做表格结构化提取
- 在 RAG 检索结果中附加 `file_path + heading_path + chunk_index`，前端渲染为可跳转链接
- 增加重排序 (Rerank) 环节提升检索精度

#### 2. Agent 执行追踪与可观测性（对标 LangSmith / Dify）

| 现状 | 目标 |
|------|------|
| 统计仪表板（Token 趋势） | Span 级 Agent 执行追踪 |
| 工具调用记录表 | 工具调用瀑布图 + 每步耗时/输入/输出可视化 |
| 无反馈闭环 | 对话质量评分 + 反馈收集 + 基于反馈优化 |

**技术路径**：
- 在 `llm_service.py` 的 Agent loop 中插桩，记录每一步的 span（tool_call 名称、参数、结果、耗时、状态）
- 前端新增 "Agent 执行追踪" 面板，瀑布图展示执行时间线
- 增加对话反馈按钮（👍/👎），存入数据库
- 统计面板增加 Agent 执行效率指标（平均步数、工具成功率、每步平均耗时）

### P1 — 能力跃升

#### 3. 多 Agent 角色协作（对标 CrewAI）

| 现状 | 目标 |
|------|------|
| system_delegate_task 层级委派 | 角色化子 Agent（role + goal + backstory） |
| 子 Agent 独立上下文，不互通 | Agent 间消息传递和争论 |
| 无协作模板 | 预置经典协作流程模板（research→write→review 链） |

**技术路径**：
- 在 `delegate.py` 中新增子 Agent 角色定义参数（role_name, goal, backstory）
- 新增协作流程模板配置（YAML/JSON 定义 Agent 角色链和消息流）
- 允许子 Agent 在完成时向主 Agent 或其他子 Agent 发送消息

#### 4. 轻量级可视化工作流编辑器（对标 Dify / LangFlow）

| 现状 | 目标 |
|------|------|
| 聊天式 Agent 自主规划 | 将已验证的 Agent 交互模式固化为可视化流程 |
| 无流程复用 | 拖拽式节点编排，条件分支，变量传递 |

**技术路径**：
- 第一步：将 Agent 的历史工具调用链可视化为流程图（只读）
- 第二步：允许用户将某次成功的 Agent 执行路径保存为"流程模板"
- 第三步：提供轻量级编辑器，允许修改节点参数和连接关系

#### 5. Skill 智能选择（对标 AnythingLLM）

| 现状 | 目标 |
|------|------|
| Skills 全量注入系统提示词 | 按语义匹配只注入相关 Skill |
| 系统提示词随 Skill 数量线性增长 | 节省 50%+ 系统提示词 token |

**技术路径**：
- 对用户最后一条消息做轻量嵌入，与每个 Skill 的 description 做相似度匹配
- Top-K 相关 Skill 注入系统提示词，其余省略
- 向下兼容：保留"全量注入"选项作为备选

### P2 — 长期优化

- **MCP 生态建设**：提供常用 MCP 服务一键配置模板（如 filesystem、web_search、database 等）
- **自动标签**：TF-IDF + TextRank 自动提取关键词为知识库标签
- **意图识别路由**：根据问题类型自动选择检索策略（精确查找 vs 语义理解）

## 八、不需追逐的领域（差异化取舍）

以下方面建议**有意识不做**，保持定位清晰：

| 领域 | 理由 |
|------|------|
| 多租户/团队协作 | LumNeo 是个人工具，不是企业平台 |
| 即时通讯多渠道发布 | 桌面应用定位，非 OpenClaw 式的消息中间件 |
| 纯零代码 | 目标用户是开发者/高级用户，保留代码能力是优势 |
| 自研模型 | 保持模型无关性，兼容所有 OpenAI 兼容 API |
| 3D 空间记忆 (Memory Palace) | 过度设计，当前向量记忆已足够好 |
| 工作流发布为 API | 桌面应用场景不需要，单机使用即可 |
| 云端 SaaS 化 | 本地隐私是核心护城河 |

## 九、总结

### 一句话定位

> **LumNeo 是目前唯一将"Obsidian 笔记 + AI RAG + Skills 技能系统 + MCP 生态 + 桌面隐私"融合的产品**——它的赛道不是成为"更好的 Dify"或"开源的 Coze"，而是成为 **"最懂你个人知识库的桌面 AI 工作台"**。
